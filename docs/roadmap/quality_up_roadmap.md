# クオリティアップ・ロードマップ

対象: ハイキングシミュレーター フェーズ2（体験品質向上）
作成日: 2026-07-08

## 1. 目指す体験（North Star）

参考画像から抽出した、このフェーズで実現したい3つの体験。

| # | 体験 | 参考イメージ | 現状とのギャップ |
|---|------|-------------|----------------|
| V1 | **稜線の遠望** — 標高が上がるほど視界が開け、幾重にも重なる山並みが空気遠近感（遠くほど青く霞む）とともにどこまでも見える | 天狗岳の稜線写真・冬山山頂パノラマ | 地形は1タイル(約1km四方)のみ。タイルの外は空。フォグ・空気遠近なし |
| V2 | **樹林帯の癒し** — 森林限界より下では、木々の間を縫う登山道を歩き、木漏れ日と緑に包まれる | シャクナゲの登山道写真 | 木は円錐プリミティブ1200本。登山道なし。地面は頂点カラーのみ |
| V3 | **重層的な音の雰囲気** — 標高・植生・天候に応じて風・鳥・沢・足音などが重なり合い、その場の空気を作る | — | 合成した風ループ1本をグローバル再生するのみ |

**体験の核となる連動**: V1とV2は「標高による遷移」で繋がる。樹林帯（視界が閉じ、鳥の声）→ 森林限界（視界が開け始める）→ 稜線（風が強まり、遠望が広がる）。この遷移を視覚と聴覚の両方で感じられることがゴール。

## 2. マイルストーン概要

依存関係順。各マイルストーンは単独でプレイアブルな改善として完結させる
（大きな作りかけ状態を持ち越さない）。

```
M1 空気と光          … フォグ・空・ライティングの基盤（全体験の土台）
 └→ M2 遠景の山並み   … マルチタイル地形で「どこまでも見える」を実現 (V1)
 └→ M3 樹林帯の質感   … 植生・登山道・地面の表現 (V2)
      └→ M4 音の重層化 … 標高・植生に連動する環境音 (V3)  ※M3の植生帯情報に依存
           └→ M5 仕上げ … 雲・天候演出・木漏れ日などの磨き込み
```

- M2とM3は並行可能（どちらもM1の上に載る）
- M4は「植生帯・森林限界の実標高化」(M3-1) にだけ依存。M3全完了を待つ必要はない

### M1: 空気と光（推定規模: 小）

視覚品質の土台。コード変更は少ないが効果が最も大きい。

| ID | タスク | 概要 |
|----|--------|------|
| M1-1 | 物理ベースの空 | `ProceduralSkyMaterial` → `PhysicalSkyMaterial` へ移行。地平線近くの霞・太陽周りの散乱を得る |
| M1-2 | 距離フォグ + 空気遠近 | `Environment` の depth フォグ + `fog_aerial_perspective` で遠景が空に溶ける表現 |
| M1-3 | 露出・トーンマップ調整 | `tonemap_mode = FILMIC/AgX`・ホワイトポイント調整。頂点カラー地形でも見栄えが変わる |
| M1-4 | 時刻連動ライティング | HUDの時計（ゲーム内時刻）に `DirectionalLight3D` の角度・色温度を連動。朝の斜光→昼→午後 |

**受け入れ基準**: 山頂から見下ろしたとき遠い谷が薄く霞む。フォールバック地形でも成立する。

### M2: 遠景の山並み — 稜線体験 (V1)（推定規模: 大）

詳細設計: [`docs/design/vista_and_atmosphere_design.md`](../design/vista_and_atmosphere_design.md)

| ID | タスク | 概要 |
|----|--------|------|
| M2-1 | 標高基準のグローバル化 | タイルごとの最小標高正規化をやめ、コース基準標高に統一（マルチタイルの前提） |
| M2-2 | 近傍タイル読み込み | プレイエリア周囲 3×3 を同ズームで取得・結合。タイル境界の継ぎ目処理 |
| M2-3 | 遠景リング | 低ズーム(z12〜13, dem10b等)の広域タイルを低解像度メッシュで周囲に配置。半径 30〜60km の山並みを描く |
| M2-4 | 描画距離とLOD | `Camera3D.far` 拡大、遠景メッシュは頂点数を落とし `visibility_range` で近景と切り替え |
| M2-5 | タイル取得基盤の強化 | 並列HTTP・リトライ・ディスクキャッシュ（GSIサーバーへの負荷配慮も含む） |

**受け入れ基準**: 山頂・稜線で四方に実在の山並みが見え、遠いほど青く霞む。標高が上がる過程で「見える範囲が広がる」ことが体感できる。

### M3: 樹林帯の質感 — 森の癒し (V2)（推定規模: 大）

詳細設計: [`docs/design/forest_experience_design.md`](../design/forest_experience_design.md)

| ID | タスク | 概要 |
|----|--------|------|
| M3-1 | 植生帯の実標高化 | 森林限界を標高比(0.75)から実標高ベース（コース緯度で可変）へ。植生帯（広葉樹/針葉樹/ハイマツ/岩礫）を定義 |
| M3-2 | 樹木アセット改善 | 帯ごとの樹形（広葉樹・針葉樹・ハイマツ）。近景はメッシュ、遠景はビルボードの2段LOD |
| M3-3 | 密度アップと風揺れ | 樹林帯の密度を体感的に「森の中」になるまで上げる（チャンク分割 + `visibility_range`）。樹冠シェーダーで風揺れ |
| M3-4 | 登山道の生成 | 登山口→山頂の最小コストパス（傾斜ペナルティ付きA*）を計算し、地面の色・プロップ除去で道を刻む |
| M3-5 | 地面の質感 | 頂点カラー → 傾斜・標高・植生帯で混合するスプラットシェーダー（土・草・岩・ガレ） |

**受け入れ基準**: 樹林帯では視界が木々で囲まれ、登山道を辿って歩ける。森林限界を超えると植生が低くなり視界が開ける遷移が分かる。

### M4: 音の重層化 (V3)（推定規模: 中）

詳細設計: [`docs/design/ambient_audio_design.md`](../design/ambient_audio_design.md)

| ID | タスク | 概要 |
|----|--------|------|
| M4-1 | オーディオバス設計 | Ambient / SFX / Weather バスとリバーブ（谷・樹林で変える） |
| M4-2 | 環境音レイヤー | 風（標高・露出度で音量/音色変化）・森（鳥/葉擦れ）・沢音のループを標高比でクロスフェード |
| M4-3 | 足音 | 地表種別（土・岩・ガレ）ごとの足音を `AudioStreamRandomizer` で再生。歩調は移動速度連動 |
| M4-4 | 点音源とワンショット | 鳥の一声・風の突風など、位置を持つ音を `AudioStreamPlayer3D` でランダム配置 |
| M4-5 | 音素材制作パイプライン | `tools/generate_ambient.py` を拡張（鳥・沢・足音の合成）。外部CC0素材の導入基準も定義 |

**受け入れ基準**: 目を閉じても「樹林帯にいるか稜線にいるか」が分かる。天候変化（雨）が音でも表現される。

### M5: 仕上げ（推定規模: 中・任意）

| ID | タスク | 概要 |
|----|--------|------|
| M5-1 | 雲 | 空マテリアルのカバー調整 or 遠景の雲ビルボード。雨イベントと連動して曇る |
| M5-2 | 木漏れ日 | 樹林帯でボリュメトリックフォグ + 影で光条を出す（Forward+専用・品質設定でON/OFF） |
| M5-3 | 天候の視覚化 | 既存の雨イベントに `GPUParticles3D` の雨・フォグ濃度上昇・空の暗転を接続 |
| M5-4 | 品質設定 | 低スペック/モバイル向けに描画距離・密度・フォグを段階設定 |

## 3. 技術スタック方針

| 領域 | 採用 | 理由 / 不採用案 |
|------|------|----------------|
| エンジン | Godot 4.6 Forward+（現状維持） | ボリュメトリックフォグ・`PhysicalSkyMaterial` 等が使える。モバイル版を出す時はMobileレンダラー用の品質フォールバック（M5-4）で対応 |
| 地形 | **自前パイプライン継続**（SurfaceTool → 将来 ArrayMesh 直組み） | GSIタイル取得・正規化・スポーン選定と密結合しており自前の方が制御しやすい。[Terrain3D](https://github.com/TokisanGames/Terrain3D) アドオンはクリップマップLODが強力だが、外部GDExtension依存とデータ形式変換のコストが大きい。M2-4で自前LODが破綻したら再検討する |
| 標高データ | 国土地理院タイル: `dem5a`(z15) 近景 / `dem10b`(z14) 中景 / 低ズーム `dem`系 遠景 | 無料・実在地形・既に統合済み。**遠景用の低ズームデータセット名と提供ズームは実装前に `tools/fetch_elevation.py` で実在確認する**（GSIのデータセット構成は変わることがある） |
| 植生・岩 | プロシージャルメッシュ（in-engine生成）+ MultiMesh | Blender製アセットは見た目は良いが制作・インポートの管理コストが高い。まずコードで樹形を組み、限界を感じたらBlender/[Tree It](https://www.evolved-software.com/treeit/treeit)等の外部モデルを検討 |
| シェーダー | Godot組み込み `gdshader`（地面スプラット・樹冠揺れ） | 依存なし。VisualShaderは差分管理しづらいのでテキストシェーダーで書く |
| 音 | 自前合成（`tools/generate_ambient.py` 拡張）を基本、不足分はCC0素材 | ライセンス管理ゼロで再現可能ビルドを維持。鳥の声など合成が難しいものだけCC0（出典は `assets/audio/CREDITS.md` に記録） |
| パスファインディング | 自前A*（グリッド直乗せ） | 256×256グリッドなら `AStarGrid2D`（Godot組み込み）で十分 |

## 4. パフォーマンス予算

60fps（16.6ms）をデスクトップ中位GPUで維持することを各マイルストーンの完了条件に含める。

| 項目 | 予算 |
|------|------|
| 地形描画（近景+遠景） | 頂点数合計 200万以下・ドローコール 30以下 |
| 植生 | MultiMesh合計インスタンス 5万以下（LOD後の描画対象はその1/4以下） |
| 環境音 | 同時再生ストリーム 12本以下 |
| タイル取得 | 起動時の同時HTTPリクエスト 4本以下・キャッシュヒット時は通信ゼロ |

## 5. リスクと対策

| リスク | 対策 |
|--------|------|
| GSIタイルの大量取得がAPI負荷・起動時間増に繋がる | ディスクキャッシュ必須(M2-5)。遠景は起動後に非同期で追加読み込みし、揃うまで近景+フォグで成立させる |
| タイル境界の段差・法線の継ぎ目 | 隣接タイルの縁1行/列を共有頂点化（設計書 M2-2 参照）。スモークテストに継ぎ目チェックを追加 |
| 植生密度アップでフレーム落ち | チャンク + `visibility_range` を密度アップと同時に入れる（後付けにしない） |
| フォールバック地形（オフライン）で新機能が壊れる | 各マイルストーンの受け入れ基準に「フォールバック地形でも動く」を含める。スモークテストはオフライン経路で実行される点を利用 |
| 頂点カラー地形とスプラットシェーダーの移行期の見た目退行 | M3-5 完了までは既存頂点カラーを残し、シェーダーはフラグで切替可能にする |

## 6. テスト方針（既存3分類への追加）

- **機能テスト**: タイル結合の座標整合（隣接タイルの縁標高一致）、A*パスが登山口と山頂を結ぶこと、植生帯の標高しきい値
- **境界値テスト**: 森林限界前後での植生・音レイヤーの切替、フォグ距離のクランプ、キャッシュ破損時のフォールバック
- **耐久テスト**: 28800フレーム歩行で音レイヤーのクロスフェードがNaN/音量爆発しないこと、ストリーミングでメモリが単調増加しないこと

## 7. 進め方

1. 各マイルストーン開始時に該当設計書を読み、変更があれば設計書を先に更新する（設計書＝一次資料）
2. 1タスク = 1 PR を基本とし、スクリーンショット（または録画）を PR に添付して見た目の変化を記録する
3. 完了したら本ロードマップの表に ✅ を付け、受け入れ基準の充足を PR 本文に書く
